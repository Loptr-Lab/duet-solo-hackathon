## 1. What This Project Is

Duet is a screen-reader-first, fully accessible 2-player chess variant, derived from a larger game called Veiled Dominion. It already exists and works today, hosted as a static build on GitHub Pages. "Solo" is the name of this specific instance/build.

**This repo's job is not to invent a new game.** It is to:
1. Move the existing Duet build onto Google Cloud (Cloud Run)
2. Add a Gemini-powered agent that runs real parts of the business (onboarding, accessibility support, and the paid-upgrade conversation) — not a chatbot bolted on for show, but something that actually replaces manual work a human would otherwise do
3. Wire up real payment processing (Stripe)
4. Produce the evidence a hackathon judge needs: logs, revenue records, real user testimonials

This is being built for the **Build with Gemini XPRIZE** hackathon (Devpost, deadline Aug 17, 2026, 1:00pm PDT), entering under the **Education & Human Potential** category.

## 2. Non-Negotiable Design Constraints

These come from Duet's original accessibility mission and must not be violated even under time pressure:

- **Screen-reader-first, not screen-reader-compatible.** Move input is via typed/spoken algebraic notation or natural-language intent (see Section 4) — never pointing, dragging, or screen-relative interaction.
- **No in-game generated audio/voice for board narration.** Screen-reader output should come from the player's own assistive technology (JAWS, NVDA, VoiceOver, TalkBack) reading properly-labeled UI, not synthesized speech generated by this app. This was originally a compliance requirement for a different accessibility jam, but it's being kept here as a design principle: relying on the player's own AT is more robust and more genuinely accessible than generating audio ourselves.
- **No hard/startling audio transients.** Any procedural audio cues (e.g., proximity tones for the Radius of Ruin mechanic) must use soft attack/decay envelopes — this is a seizure/sensory-safety requirement, not a style preference.
- **The Gemini agent's job is text-in, structured-query-or-action, text-out.** It should never generate its own audio output — see above.

## 4. The Gemini Agent — Scope

Build one agent with three responsibilities, in this priority order:

1. **Onboarding:** greets a new player, walks them through initial setup (confirms their screen reader is detected/working, explains move input format), answers plain-language questions about how the game works
2. **Accessibility support:** answers questions about the Radius of Ruin / Sanctuary mechanics, move notation, or anything a confused new player might ask
3. **Upsell:** when appropriate (not pushy — the accessibility mission comes first), tells the player about the paid "Accessibility Pack" (custom audio cue themes, saved profile, supporter badge) and hands off to the Stripe checkout link

**Every agent interaction must be logged** — this is required hackathon evidence ("agent execution logs, API usage records"), not optional instrumentation. Log format should capture: timestamp, user input, agent's interpreted intent, agent's action/response, and outcome.

**Do not build a move-generation or move-suggestion AI.** The agent's job is business operations (onboarding, support, sales) — it is not meant to play chess or give strategic advice. Keep scope tight.
